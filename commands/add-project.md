description: "Add a project to the Fang agent. Import an existing repo or initialise a new one from scratch. Updates ~/fang/projects.json and bootstraps the project for autonomous operation."
---

You are Fang. The user wants to add a project. Parse `$ARGUMENTS` to determine the sub-command and options, then execute the matching flow below.

---

## Argument parsing

`$ARGUMENTS` will be one of:

```
import <repo-url> [--name <custom-name>]
init <project-name> [--description "<what it does>"]
```

- **`import`** — clone an existing remote repository and wire it up
- **`init`** — create a brand-new project from scratch

If neither keyword is present, send a WhatsApp message to the owner explaining correct usage and stop.

Parse the arguments now. Set:

- `MODE` = `import` or `init`
- `REPO_URL` = the URL (import only)
- `PROJECT_NAME` = custom name if `--name` was given, otherwise the last path component of the URL with `.git` stripped (import), or the bare name given (init)
- `DESCRIPTION` = value of `--description` if given, otherwise `""`

Validate: `PROJECT_NAME` must match `[a-z0-9_-]+`. If it does not, notify owner and stop.

---

## Shared pre-flight checks

Before running either flow, verify:

1. `~/fang/projects.json` exists and is valid JSON.
2. No entry in `projects.json` already has `"name": "<PROJECT_NAME>"`. If a duplicate exists, notify owner and stop.
3. `~/projects/` directory exists:
   ```sh
   mkdir -p ~/projects
   ```
4. A tmux slot is available (capacity guard — max 2 concurrent sessions):
   ```sh
   ACTIVE=$(tmux list-sessions 2>/dev/null | grep -cE "^(proj-|forge-|inv-)" || echo 0)
   ```
   If `ACTIVE >= 2`, queue this request as a `workflow` belief and notify owner:
   ```sh
   mem-reason add-belief \
     --text "Queued: add-project $MODE $PROJECT_NAME — waiting for free slot" \
     --domain "workflow" --confidence 0.9 --tags "queued-task,add-project"
   ```
   Then stop. The heartbeat will dequeue it when a slot opens.

## Flow A — Import existing repo

### A1. Clone

```sh
git clone "$REPO_URL" ~/projects/"$PROJECT_NAME"
```

If the clone fails (non-zero exit), record a belief and notify owner:
```sh
mem-reason add-belief \
  --text "Clone of $REPO_URL into ~/projects/$PROJECT_NAME failed" \
  --domain "project" --confidence 0.9 --tags "clone-failed,$PROJECT_NAME"
wa send <owner_jid> "add-project import failed: could not clone $REPO_URL. Check SSH key and repo URL."
```
Stop.

### A2. Detect package manager and install dependencies

```sh
cd ~/projects/"$PROJECT_NAME"

if [ -f "bun.lockb" ] || [ -f "bunfig.toml" ]; then
  bun install
elif [ -f "yarn.lock" ]; then
  yarn install --frozen-lockfile
elif [ -f "pnpm-lock.yaml" ]; then
  pnpm install --frozen-lockfile
elif [ -f "package.json" ]; then
  npm install
elif [ -f "requirements.txt" ]; then
  python3 -m pip install -r requirements.txt
elif [ -f "Pipfile" ]; then
  pipenv install
elif [ -f "Gemfile" ]; then
  bundle install
elif [ -f "go.mod" ]; then
  go mod download
elif [ -f "Cargo.toml" ]; then
  cargo fetch
else
  echo "No recognised dependency manifest. Skipping install."
fi
```

If install fails, record a belief noting the failure but continue (project can still be registered).

### A3. Ensure CLAUDE.md exists

```sh
if [ ! -f ~/projects/"$PROJECT_NAME"/CLAUDE.md ]; then
  if [ -f ~/ccrice/templates/CLAUDE-TEMPLATE.md ]; then
    cp ~/ccrice/templates/CLAUDE-TEMPLATE.md ~/projects/"$PROJECT_NAME"/CLAUDE.md
    sed -i "s/{{PROJECT_NAME}}/$PROJECT_NAME/g" ~/projects/"$PROJECT_NAME"/CLAUDE.md
  else
    cat > ~/projects/"$PROJECT_NAME"/CLAUDE.md << 'CLAUDEMD'
# PROJECT_NAME_PLACEHOLDER

> Auto-generated by Fang add-project. Update this file with project-specific details.

## Closed Dev Loop

All changes follow this sequence — no exceptions:

1. **Implement** — write or modify code to satisfy the task
2. **Review** — static analysis, lint, type-check (`npm run lint` / equivalent)
3. **Fix** — address any issues found in review before proceeding
4. **Test** — run the test suite; all tests must pass
5. **Deploy** — only after tests pass; staging first, then prod with approval

Never skip a step. If a step cannot complete (no test suite, no lint config), record that as a belief and notify the master agent.

## Reporting

Every session must write a JSON report to `REPORT_TO` before exiting. Use the schema in `~/fang/report-schema.json`.
CLAUDEMD

    sed -i "s/PROJECT_NAME_PLACEHOLDER/$PROJECT_NAME/g" ~/projects/"$PROJECT_NAME"/CLAUDE.md
  fi
fi
```

> **Note:** The CLAUDE.md has been scaffolded from the template. Placeholders marked with `{{...}}` need to be filled in for the project's specific stack.

### A4. Initialise memr

```sh
cd ~/projects/"$PROJECT_NAME" && mem-reason init
```

If `mem-reason init` fails, record a belief and continue — the project can operate without local memory (it will use master memory only).

### A5. Register in projects.json

Read `~/fang/projects.json`, append the new entry, write it back. The new entry shape:

```json
{
  "name": "<PROJECT_NAME>",
  "repo": "<REPO_URL>",
  "description": "<DESCRIPTION or empty string>",
  "deploy_script": "deploy-<PROJECT_NAME>.sh",
  "deploy_envs": ["staging", "prod"],
  "watchers": [],
  "added": "<YYYY-MM-DD>",
  "status": "active"
}
```

Use the current UTC date for `added`. Use the actual deploy script name if detectable from the repo root (`ls deploy-*.sh`); otherwise default to `deploy-<PROJECT_NAME>.sh`.

### A6. Record belief

```sh
mem-reason add-belief \
  --text "Added project $PROJECT_NAME from $REPO_URL on $(date -u +%Y-%m-%dT%H:%M:%SZ)" \
  --domain "project" --confidence 0.9 --tags "project-registered,$PROJECT_NAME"
```

### A7. Notify owner

```sh
wa send <owner_jid> "Project '$PROJECT_NAME' imported successfully from $REPO_URL. Dependencies installed. Registered in projects.json. Ready to use."
```

## Flow B — Init new project from scratch

### B1. Create directory and git repo

```sh
mkdir -p ~/projects/"$PROJECT_NAME"
cd ~/projects/"$PROJECT_NAME" && git init
```

### B2. Create CLAUDE.md

```sh
if [ -f ~/ccrice/templates/CLAUDE-TEMPLATE.md ]; then
  cp ~/ccrice/templates/CLAUDE-TEMPLATE.md ~/projects/"$PROJECT_NAME"/CLAUDE.md
  sed -i "s/{{PROJECT_NAME}}/$PROJECT_NAME/g" ~/projects/"$PROJECT_NAME"/CLAUDE.md
else
  cat > ~/projects/"$PROJECT_NAME"/CLAUDE.md << 'CLAUDEMD'
# PROJECT_NAME_PLACEHOLDER

> Auto-generated by Fang add-project. Update this file with project-specific details.

## Closed Dev Loop

All changes follow this sequence — no exceptions:

1. **Implement** — write or modify code to satisfy the task
2. **Review** — static analysis, lint, type-check (`npm run lint` / equivalent)
3. **Fix** — address any issues found in review before proceeding
4. **Test** — run the test suite; all tests must pass
5. **Deploy** — only after tests pass; staging first, then prod with approval

Never skip a step. If a step cannot complete (no test suite, no lint config), record that as a belief and notify the master agent.

## Reporting

Every session must write a JSON report to `REPORT_TO` before exiting. Use the schema in `~/fang/report-schema.json`.
CLAUDEMD

  sed -i "s/PROJECT_NAME_PLACEHOLDER/$PROJECT_NAME/g" ~/projects/"$PROJECT_NAME"/CLAUDE.md
fi
```

> **Note:** The CLAUDE.md has been scaffolded from the template. Placeholders marked with `{{...}}` need to be filled in for the project's specific stack.

### B3. Ask owner about stack via WhatsApp

Send a targeted question — do not proceed to create package.json until answered:

```sh
wa send <owner_jid> "Project '$PROJECT_NAME' initialised locally. What stack should I set up? (e.g. 'Node/Express', 'Next.js', 'Python/FastAPI', 'bare Node', etc.)"
```

Store a workflow belief to resume when the owner replies:

```sh
mem-reason add-belief \
  --text "Awaiting stack choice from owner for new project $PROJECT_NAME" \
  --domain "workflow" --confidence 0.95 \
  --tags "awaiting-input,add-project,$PROJECT_NAME"
```

When the owner replies with a stack choice (detected on a subsequent heartbeat):
- Clear the `awaiting-input` belief for this project
- Generate appropriate scaffold files:
  - **Node/Express or bare Node**: `package.json` with `"main": "index.js"`, `"scripts": {"start": "node index.js", "test": "node --test"}`
  - **Next.js**: `package.json` with Next.js deps, `pages/index.jsx` stub
  - **Python/FastAPI**: `requirements.txt` with `fastapi uvicorn`, `main.py` stub
  - **Python/Flask**: `requirements.txt` with `flask`, `app.py` stub
  - **Other**: ask owner for a minimal manifest, store it, continue

### B4. Initialise memr

```sh
cd ~/projects/"$PROJECT_NAME" && mem-reason init
```

### B5. Register in projects.json

Same shape as Flow A §A5, but with `"status": "active"` and `"repo": ""` (no remote yet).

### B6. Optionally create GitHub repo

Ask owner before creating a remote:

```sh
wa send <owner_jid> "Should I create a private GitHub repo for '$PROJECT_NAME' and push the initial commit? Reply 'yes' or 'no'."
```

Store a workflow belief:
```sh
mem-reason add-belief \
  --text "Awaiting GitHub repo creation approval for $PROJECT_NAME" \
  --domain "workflow" --confidence 0.95 \
  --tags "awaiting-input,github-create,$PROJECT_NAME"
```

On owner approval (detected on next heartbeat):
```sh
cd ~/projects/"$PROJECT_NAME"
git add -A
git commit -m "chore: initial scaffold"
gh repo create "$PROJECT_NAME" --private --source . --push
```

Update `projects.json` to set `"repo": "git@github.com:kream0/<PROJECT_NAME>.git"` and `"status": "active"`.

Update the project belief:
```sh
mem-reason add-belief \
  --text "GitHub repo created for $PROJECT_NAME at git@github.com:kream0/$PROJECT_NAME.git" \
  --domain "project" --confidence 0.9 --tags "github,$PROJECT_NAME"
```

### B7. Record belief and notify

```sh
mem-reason add-belief \
  --text "Initialised new project $PROJECT_NAME on $(date -u +%Y-%m-%dT%H:%M:%SZ)" \
  --domain "project" --confidence 0.9 --tags "project-registered,$PROJECT_NAME"

wa send <owner_jid> "Project '$PROJECT_NAME' is ready. CLAUDE.md created with closed dev loop rules. Registered in projects.json."
```

## Error reference

| Error | Action |
|---|---|
| `git clone` fails | Record belief, notify owner, stop |
| Duplicate project name in projects.json | Notify owner, stop |
| `projects.json` malformed or missing | Notify owner: "projects.json is missing or corrupt — fix manually before adding projects." Stop. |
| `mem-reason init` fails | Record belief, continue (not fatal) |
| `gh repo create` fails | Record belief, notify owner, continue without remote |
| Install step fails | Record belief noting partial install, continue |
